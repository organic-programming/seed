## Phase 0 — Preparation (Codex, single session, before any SDK loop)

### Codex prompt — Phase 0

```text
You are working in the organic-programming monorepo.

Goal: Extend `op build` to generate Incode Description source files from templates.

Context — read these first:
- sdk/README.md § "Incode Description" — the feature spec
- sdk/go-holons/pkg/describe/describe.go — runtime Describe (parses holon.proto + proto/ at startup via protoreflect)
- sdk/go-holons/pkg/identity/resolve.go — manifest resolver (ProtoManifestFileName = "holon.proto")
- holons/grace-op/internal/ — op internals (cli/, do/, identity/, manifestproto/)
- holons/grace-op/HOLON_BUILD.md — current op build spec

New feature — "proto stage" in op build:
1. After manifest validation and before the language-specific build step, add a stage that:
   a. Resolves holon.proto + proto/ using the same logic as go-holons/pkg/describe.BuildResponse().
   b. Determines the SDK language from the manifest (lang field).
   c. Looks for a Go text/template at sdk/<lang>-holons/templates/describe.<ext>.tmpl.
   d. Executes the template with the parsed DescribeResponse data model.
   e. Writes the output to gen/describe_generated.<ext> in the holon source tree.
2. If no template exists for the language, skip silently (the SDK hasn't opted in yet).
3. If the template exists but rendering fails, the build must fail with a clear error.

Reference template — create sdk/go-holons/templates/describe.go.tmpl:
- The template produces a Go source file that exports a function:
  func StaticDescribeResponse() *holonsv1.DescribeResponse
- The function returns a compile-time static response with identity, services, and RPCs.
- The generated file should have a "Code generated by op build. DO NOT EDIT." header.

Tests:
- Write unit tests for the template rendering in holons/grace-op.
- Write a test that generates from sdk/go-holons/templates/describe.go.tmpl with sample data and verifies the output compiles.

Commits: use conventional commits prefixed with feat(grace-op):.
Do NOT modify any SDK other than go-holons.
```

## Phase 1 — Per-SDK Loop

**Strategy**: Run Go first (it's the reference). Once Go is done, launch all 12 remaining SDKs **in parallel**.

### Codex prompt — Per-SDK

> Before pasting, replace only the three variables at the top (`LANG`, `SDK_DIR`, `EXAMPLE_DIR`).

```text
You are working in the organic-programming monorepo.

LANG     = go
SDK_DIR  = sdk/go-holons
EXAMPLE_DIR = examples/hello-world/gabriel-greeting-go

Reference implementation (completed): sdk/go-holons — use it as the canonical pattern for every step.

Context files to read first:
- sdk/README.md — Incode Description spec, transport matrix, per-SDK README spec
- sdk/go-holons/ — reference implementation for all modules

Execute the following steps IN ORDER. After each step, commit with a conventional commit message prefixed with `fix(LANG-holons):` or `feat(LANG-holons):`.

### Step 1 — Incode Description Template
Create `SDK_DIR/templates/describe.<ext>.tmpl`.
This Go template receives the same data model as the Go reference and produces a native source file that exports a static DescribeResponse equivalent.

### Step 2 — Explicit Describe Registration Errors
In the SDK's serve module, find where the Describe (HolonMeta) endpoint is registered.
If registration can fail silently (error swallowed, nil check missing, etc.), make the error explicit: log it clearly and propagate it so the holon fails to start.

### Step 3 — Standard Describe / Identity / Discover
Review and align (or create) the SDK's `describe`, `identity`, and `discover` modules to match the go-holons pattern:
- `describe`: Should work with the generated Incode Description (step 1) AND as a fallback with runtime proto parsing.
- `identity`: Must read `holon.proto` and expose the resolved manifest.
- `discover`: Must scan the filesystem for nearby holons by slug.

### Step 4 — Proto-less Describe
Ensure Describe works on an isolated binary that has NO .proto files alongside it.
When using the generated Incode Description, Describe must return the static response without any file I/O.

### Step 5 — Transport Matrix Audit
For each transport in the matrix (`tcp://`, `unix://`, `stdio://`, `ws://`, `wss://`, `rest+sse://`):
- Read the source code to determine if Dial and/or Serve is implemented.
- Write or verify a test that proves the capability.
- Update sdk/README.md's Transport Matrix row for this SDK replacing `?` with the proven value (`both`, `dial`, `serve`, or `-`).
Do NOT assume — only mark capabilities you can prove with a passing test.

### Step 6 — Implement Required Transports
Compare the current state (step 5) with the expected v0.6 matrix in sdk/README.md.
Implement any MISSING transports that are required. Each new transport must have a test.

### Step 7 — Test Suite Cleanup
Run the full existing test suite for this SDK.
- Fix any broken tests.
- Remove tests that are no longer relevant.
- Ensure all tests pass.

### Step 8 — Rebuild Hello-World Example
Using the SDK's Incode Description template, rebuild EXAMPLE_DIR via `op build`.
Fix any issues that surface.

### Step 9 — Per-SDK README
Create or rewrite `SDK_DIR/README.md` following the structure defined in sdk/README.md § "Per-SDK Documentation":
- serve — one working snippet
- transport — how to specify the listen URI
- identity / describe — one line to wire Incode Description
- discover — one call to find a holon
- connect — one call to get a ready channel

### Step 10 — Final Check
Run all tests. Ensure no regressions.
```
