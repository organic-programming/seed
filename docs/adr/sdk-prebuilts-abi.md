# ADR: SDK Prebuilts Artifact ABI Policy

Status: Proposed

Date: 2026-04-26

## Context

This ADR governs the artifact ABI policy for SDK prebuilt archives. It does not
define the public language API of each SDK. For Zig's public C ABI, see
`docs/adr/zig-sdk-abi.md`.

The v1 archive scope is exactly:

- `ruby`
- `c`
- `cpp`
- `zig`

Python and T2 mobile targets are out of scope for v1.

## Artifact Identity

Every artifact identity includes:

- SDK name;
- SDK or upstream version;
- target triplet;
- archive format.

The v1 release streams are:

```text
zig-holons-v0.1.0
cpp-holons-v1.80.0
c-holons-v1.80.0
ruby-holons-v1.58.3
```

The v1 targets are T0 + T1 only:

- `aarch64-apple-darwin`
- `x86_64-apple-darwin`
- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`
- `x86_64-unknown-linux-musl`
- `aarch64-unknown-linux-musl`
- `x86_64-windows-gnu`

`x86_64-pc-windows-msvc` remains the v1.x Windows follow-up once winwok is
deployed and native MSVC builds are available.

## SDK-Specific ABI Boundaries

For `c` and `cpp`, archives contain static native dependency bundles built from
gRPC `1.80.0`, protobuf-c `1.5.2`, and their transitive dependencies. The C++
ABI is tied to the target runner compiler and runtime, including libstdc++ or
MSVC where applicable. Consumers are promised compatibility only with the
matching SDK version, target triplet, and SDK build integration.

For `ruby`, archives contain a vendored `vendor/bundle/` tree for the Ruby gem
platform. Compatibility is limited to the matching Ruby ABI/platform and pinned
gem versions, especially `grpc` `1.58.3` and `google-protobuf`. The workflow
builds these archives with Ruby 3.1; consumers with a different Ruby ABI must
fall back to Bundler until a matching Ruby prebuilt stream is added.

For `zig`, archives contain the SDK static library plus gRPC and protobuf-c
native dependencies. The public C ABI is governed by
`docs/adr/zig-sdk-abi.md`; the prebuilt archive version tracks the Zig SDK
release stream.

## Protoc-Generated Code

For each native SDK that ships protoc-generated code, protoc version is
anchored on the prebuilt. Committed gencode in `sdk/<lang>-holons/gen/` MUST be
produced by the protoc shipped in the corresponding prebuilt. Bumping the
prebuilt's protobuf version requires regenerating committed gencode in the same
PR.

## Bundled Third-Party Headers

Small header-only third-party dependencies that the SDK needs at compile time
are bundled in the corresponding prebuilt when doing so keeps the prebuilt
self-contained without adding a native ABI surface. The C++ prebuilt bundles
`nlohmann/json` `3.11.3` under `include/nlohmann/json.hpp`. The C prebuilt
also bundles that header because the current C SDK serve bridge includes the C++
SDK bridge header. Larger dependencies or dependencies with compiled artifacts
remain part of the native dependency bundle or the documented system fallback,
case by case.

## Versioning

Semver applies per artifact stream.

An incompatible archive layout or ABI change requires a major version bump once
that stream is stable. During `0.x`, incompatible changes are allowed, but they
must be visible in the release manifest and documented in the release notes.

Patch rebuilds must not silently replace published release assets. A rebuild
requires either a new release tag or explicit rebuild metadata that
distinguishes the artifact in the manifest.

## Soname and Shared Libraries

V1 prefers static archives. If shared libraries are added later, every shared
library must set the platform ABI marker:

- Linux: `soname` matching the ABI major;
- macOS: `install_name` matching the ABI major;
- Windows: DLL naming and import-library metadata matching the ABI major.

Major ABI breaks require both a shared-library ABI marker bump and a release tag
bump.

## Stripping and Debug Sidecars

Stripped main archive + sidecar debug archive per platform: .dSYM/ (macOS),
.pdb (Windows), -debug.tar.gz (Linux).

The main archive is stripped. The debug sidecar carries symbols and must share
the same manifest identity as the main archive. Release manifests record
separate SHA-256 hashes for main archives and debug sidecars.

## Integrity and SBOM

V0.x ships SHA-256 verification only. Cosign keyless signing is added at v1.0.

Production artifacts ship SPDX SBOMs generated by the workflow. M0 used stub
SPDX JSON files only to validate packaging shape.

Each artifact manifest records at least:

- SDK name;
- SDK or upstream version;
- target triplet;
- archive SHA-256;
- debug sidecar SHA-256 when present;
- SBOM URL and SHA-256;
- source pins;
- build runner or image;
- toolchain versions;
- ABI metadata needed by the SDK build integration.
