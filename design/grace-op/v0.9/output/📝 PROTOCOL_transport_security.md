## 8. Transport Security

### 8.1 Threat Model

Holon-to-holon communication across hosts faces three threats:

| Threat | Description | Mitigation |
|---|---|---|
| **Eavesdropping** | Passive observer reads traffic | TLS encryption |
| **Man-in-the-Middle** | Active attacker intercepts and modifies traffic | Certificate verification |
| **Impersonation** | Attacker pretends to be a legitimate holon | Mutual authentication (mTLS) |

Local transports (`stdio://`, `unix://`, `mem://`) are not exposed
to these threats — they never leave the host.

### 8.2 mTLS for gRPC

For holon-to-holon communication across hosts, the mesh uses **mutual
TLS** (mTLS): both caller and callee present certificates signed by
the mesh CA.

| Parameter | Value |
|---|---|
| Protocol | TLS 1.3 |
| Cipher | AES-256-GCM (negotiated) |
| CA | Mesh-local ECDSA P-256 CA (generated by `op mesh init`) |
| Server cert | Per-host, signed by mesh CA |
| Client cert | Per-host, signed by mesh CA |
| Verification | `RequireAndVerifyClientCert` (both directions) |

### 8.3 Certificate Identity

Each mesh certificate carries identity in two fields:

- **CN (Common Name)**: hostname (e.g., `lyon.example.com`)
- **SAN (Subject Alternative Names)**: DNS name + IP address

The SDK extracts the peer's identity from the verified certificate:

```go
// Extract peer hostname from mTLS connection
peer := auth.PeerFromContext(ctx)
hostname := peer.CN  // "lyon.example.com"
```

### 8.4 Local Transports — No TLS

| Transport | Security | Reason |
|---|---|---|
| `stdio://` | None | Pipes between parent and child process |
| `unix://` | None | Unix socket, same host, filesystem ACL |
| `mem://` | None | In-process channel, no network exposure |

These transports are inherently confined to the local host. TLS
would add overhead without security benefit.

### 8.5 Per-Listener Security Modes

Each listener in `holon.yaml` carries a `security` annotation:

| Mode | Encryption | Authentication | Use case |
|---|---|---|---|
| `none` | None | None | Local transports |
| `mesh` | mTLS (TLS 1.3) | Mutual certs from mesh CA | Holon-to-holon across hosts |
| `public` | TLS (standard) | API key / JWT / OAuth | Internet-facing API |

```yaml
serve:
  listeners:
    - uri: tcp://:9090
      security: mesh
    - uri: tcp://:443
      security: public
      auth: api-key
    - uri: unix:///tmp/holon.sock
      security: none
```

Auto-detection defaults:
- `stdio://`, `unix://`, `mem://` → `none` (always)
- `tcp://` with mesh certs present → `mesh`
- `tcp://` without mesh certs → `none` (with warning)

### 8.6 SDK Integration

#### `serve.Run` auto-detection

When a holon starts via `serve.Run`, the SDK checks for mesh
certificates in `~/.op/mesh/`. If found:
- TCP listeners automatically use mTLS
- `RequireAndVerifyClientCert` is enforced
- No configuration needed in `holon.yaml`

#### `connect` mTLS

When connecting to a remote holon (discovered via mesh registry),
the SDK loads the local host's cert and the CA cert:

```
connect("phill-files") on remote host:
  1. discover → mesh.yaml → lyon.example.com:9090
  2. Load ~/.op/mesh/{host.key, host.crt, ca.crt}
  3. Dial with mTLS credentials
  4. Verify server cert against mesh CA
```

### 8.7 Future: Application-Level E2EE

The current security model protects the transport. For scenarios
requiring end-to-end encryption (where even the transport endpoint
must not see plaintext), application-level encryption can be layered
on top. This is out of scope for the mesh transport layer.

### Addition to §2 — Remote Discovery via Mesh Registry

> Insert after current §2.4 "Discover" in PROTOCOL.md.

When a holon is not found locally, `discover` consults the mesh
registry:

```
discover("phill-files") search order:
1. Local OPPATH scan (existing behavior)
2. Read ~/.op/mesh/mesh.yaml
3. For each host, query HolonMeta/Describe over mTLS (cached)
4. Return matching holon with remote address + port
```

The mesh registry extends discovery to remote hosts without
changing the `discover` API. Callers get a remote address instead
of a local path — the `connect` layer handles the rest.
